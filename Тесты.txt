# 13. Тестирование

Данный раздел определяет перечень тестов, которые должны быть проведены на каждом этапе разработки для обеспечения качества и соответствия требованиям. Тесты делятся на уровни и функциональные области.

## 13.1 Уровни тестирования

### 13.1.1 Модульное тестирование
Проводится разработчиком в процессе кодирования каждого модуля. Цель – проверить корректность работы отдельных функций, классов, методов.

**Требования:**
- Покрытие ключевой бизнес-логики модульными тестами не менее 70%.
- Использование фреймворков (например, pytest для Python, JUnit для Java, Jest для JavaScript).
- Автоматический запуск тестов при CI/CD.

### 13.1.2 Интеграционное тестирование
Проверка взаимодействия между модулями (например, связь бэкенда с БД, работа API, взаимодействие фронтенда с бэкендом).

**Требования:**
- Тестирование всех API-эндпоинтов (позитивные и негативные сценарии).
- Проверка корректности передачи данных между модулями.
- Тестирование внешних интеграций (FCM, СМС-шлюз, Telegram Bot API).

### 13.1.3 Системное тестирование (End-to-End)
Проверка системы в целом на соответствие функциональным требованиям. Выполняется тестировщиком по заранее подготовленным тест-кейсам.

**Требования:**
- Покрытие всех пользовательских сценариев.
- Тестирование на реальных данных.
- Проверка работы системы под нагрузкой, близкой к реальной.

### 13.1.4 Тестирование безопасности
Оценка устойчивости системы к типовым атакам.

**Требования:**
- Проверка на SQL-инъекции, XSS, CSRF.
- Анализ управления сессиями и аутентификацией.
- Проверка прав доступа (ролевая модель).

### 13.1.5 Тестирование производительности
Проверка времени отклика и стабильности при одновременной работе нескольких пользователей.

**Требования:**
- Тестирование с имитацией 10–20 одновременных сессий сотрудников и до 100 клиентов.
- Замер времени ответа API (не более 500 мс).
- Проверка стабильности при длительной работе (сутки).

### 13.1.6 Тестирование удобства использования (UI/UX)
Оценка интерфейса на соответствие требованиям, интуитивность, удобство навигации.

**Требования:**
- Проверка на соответствие макетам.
- Тестирование на мобильных устройствах (адаптивность).
- Опрос пользователей (по возможности).

### 13.1.7 Приемочное тестирование
Проводится заказчиком после завершения разработки. Проверка соответствия системы ожиданиям и готовности к эксплуатации.

**Требования:**
- Выполнение всех критических сценариев.
- Отсутствие блокирующих ошибок.
- Наличие документации.

## 13.3 Порядок проведения тестирования

1. **Модульное тестирование** – выполняется разработчиком в процессе написания кода.
2. **Интеграционное тестирование** – после завершения разработки каждого модуля, но до объединения в систему.
3. **Системное тестирование** – после завершения всей разработки, перед передачей заказчику.
4. **Тестирование безопасности и производительности** – проводится параллельно с системным тестированием.
5. **Приемочное тестирование** – проводится заказчиком после успешного прохождения предыдущих этапов.

Все найденные ошибки фиксируются в системе отслеживания (issue tracker), после исправления проводится регрессионное тестирование затронутой функциональности.
