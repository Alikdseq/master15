Анализ уязвимостей и защита CRM-системы «Мастер Принт»
1. Введение
Безопасность CRM — критически важный аспект, так как система хранит персональные данные клиентов (ФИО, телефоны, адреса), информацию о заказах, финансовые данные (стоимость, себестоимость), учётные данные сотрудников. Любая уязвимость может привести к утечке конфиденциальной информации, потере репутации или даже финансовым потерям.

Ниже приведён структурированный анализ угроз и рекомендации по защите на всех этапах жизненного цикла системы.

2. Данные, подлежащие защите
Категория	Примеры данных	Уровень критичности
Персональные данные клиентов	ФИО, телефон, email, адрес, история заказов	🔴 Высокий (подпадает под 152-ФЗ)
Данные сотрудников	Логин, email, хеш пароля, телефон	🔴 Высокий
Финансовые данные	Предварительная/окончательная стоимость, себестоимость материалов, итоговая прибыль	🟠 Средний
Коммерческая тайна	Складские остатки, закупочные цены, список поставщиков	🟠 Средний
Системные настройки	API-ключи SMS-провайдера, параметры подключения к БД	🔴 Высокий
Логи операций	Действия пользователей, история изменений	🟡 Низкий (но важны для аудита)
3. Потенциальные угрозы и атаки
3.1 Аутентификация и авторизация
Угроза	Описание	Возможные последствия
Подбор пароля (brute-force)	Злоумышленник перебирает пароли к учётным записям	Несанкционированный доступ
Слабая политика паролей	Использование простых паролей (123456, qwerty)	Легкость подбора
Отсутствие двухфакторной аутентификации (2FA)	Только пароль — недостаточно	Компрометация при утечке пароля
Неверное разделение ролей	Мастер может получить доступ к админ-панели	Повышение привилегий
Session hijacking	Перехват сессионной cookie (не защищена флагом HttpOnly, Secure)	Вход под чужим аккаунтом
3.2 Инъекции
Угроза	Описание	Возможные последствия
SQL-инъекция	Вставка вредоносного SQL-кода в поля ввода	Чтение/изменение/удаление данных БД
NoSQL-инъекция	Аналогично для MongoDB (если используется)	Полная компрометация БД
3.3 XSS (межсайтовый скриптинг)
Угроза	Описание	Возможные последствия
Reflected XSS	Внедрение скрипта через параметры URL	Кража сессии, редирект
Stored XSS	Сохранение вредоносного кода в полях (имя клиента, комментарий)	Заражение всех пользователей, просматривающих данные
3.4 CSRF (межсайтовая подделка запроса)
Угроза	Описание	Возможные последствия
Отсутствие CSRF-токенов	Злоумышленник может заставить авторизованного пользователя выполнить действие	Несанкционированное изменение данных
3.5 Уязвимости API
Угроза	Описание	Возможные последствия
Отсутствие rate limiting	Нет ограничения на количество запросов	Brute-force, DoS
Недостаточная валидация входных данных	Приём больших объёмов, неверных типов	Переполнение, ошибки
Отсутствие защиты от подделки JWT	Слабый секрет, нет проверки подписи	Подделка токена
Инъекции в параметры	Передача SQL-кода через параметры запроса	SQL-инъекция
3.6 Хранение данных
Угроза	Описание	Возможные последствия
Пароли в открытом виде	Хранение паролей без хеширования	Полная компрометация учётных записей
Недостаточная защита API-ключей	Ключи SMS-провайдера в коде или не в .env	Использование ключей злоумышленником
Утечка дампа БД	Несанкционированный доступ к файлу БД	Массовая утечка всех данных
Отсутствие шифрования чувствительных полей	Телефоны, адреса в открытом виде	Нарушение 152-ФЗ
3.7 Инфраструктурные атаки
Угроза	Описание	Возможные последствия
DDoS	Перегрузка сервера запросами	Недоступность системы
Уязвимости сервера (OS, веб-сервер, СУБД)	Необновлённое ПО	Внедрение вредоносного кода
Открытые порты	Доступность административных портов из интернета	Прямой доступ к БД, SSH
Отсутствие HTTPS	Передача данных в открытом виде	Перехват паролей, сессий
3.8 Атаки на бизнес-логику
Угроза	Описание	Возможные последствия
Массовое списание материалов	Злоумышленник может добавить материалы без проверки	Искажение складского учёта
Несанкционированная смена статуса	Мастер может завершить заказ, который не ремонтировал	Потеря выручки, хаос
Удаление заказов без проверки	Администратор может случайно удалить завершённый заказ	Потеря данных для отчётности
3.9 Атаки через внешние интеграции
Угроза	Описание	Возможные последствия
Перехват API-запросов к SMS-провайдеру	Если используется HTTP без шифрования	Просмотр текстов SMS, подмена отправителя
Подделка ответа от SMS-шлюза	Если не проверяется подпись	Ложная фиксация отправки
3.10 Социальная инженерия
Угроза	Описание	Возможные последствия
Фишинг	Злоумышленник выманивает пароль у сотрудника	Компрометация учётной записи
Спуфинг	Подделка SMS от имени сервиса	Обман клиентов
4. Чек-лист мер защиты по этапам
4.1 На этапе проектирования и разработки
Аутентификация и авторизация
Хеширование паролей: использовать bcrypt (стоимость 10+) или Argon2.

Политика паролей: минимум 8 символов, буквы + цифры + спецсимволы.

Ограничение попыток входа: после 5 неудачных попыток блокировка на 15 минут (rate limiting на уровне приложения).

JWT: короткое время жизни (15–30 минут), refresh token, подпись HS256 с секретом в .env.

2FA (опционально): для администраторов можно добавить через TOTP (Google Authenticator).

Ролевая модель: строгое разделение прав на уровне API (middleware проверяет роль).

Защита от инъекций
Использование ORM/параметризованных запросов (SQLAlchemy, Django ORM, или ручная параметризация).

Экранирование ввода: все данные, подставляемые в запросы, проходят через экранирование.

Валидация входных данных: типы, длина, формат (email, телефон). Использовать библиотеки (например, Pydantic, Marshmallow).

Защита от XSS и CSRF
Экранирование вывода: при отображении данных в HTML использовать безопасные шаблоны (например, Jinja2 autoescape).

Content Security Policy (CSP): запретить inline-скрипты, ограничить источники.

CSRF-токены: для всех мутирующих запросов (POST/PUT/DELETE) использовать CSRF-токен (в веб-интерфейсе). Для API использовать JWT, который сам по себе защищает от CSRF.

Безопасность API
Rate limiting: для всех эндпоинтов (особенно аутентификации, создания заказов).

Валидация JSON-схемы: отклонять запросы с лишними полями.

CORS: разрешён только доверенный домен (например, фронтенд на том же домене).

Защита от массового присваивания: разрешать обновление только разрешённых полей.

Хранение данных
Шифрование чувствительных полей: телефоны, адреса, email можно хранить в зашифрованном виде (AES-256). Ключ шифрования в .env.

Снапшоты цен: использовать отдельные поля для сохранения цен на момент заказа (как описано в ТЗ).

Логирование действий: фиксировать все изменения, но не хранить пароли.

4.2 На этапе настройки инфраструктуры
Сервер и сеть
Использование HTTPS: обязательный сертификат (Let's Encrypt), перенаправление HTTP→HTTPS.

Файрвол: разрешены только порты 80, 443 (и SSH для администрирования с ограничением по IP).

Регулярные обновления: ОС, веб-сервер (Nginx), база данных, язык программирования.

Изоляция: приложение не работает от root, своя БД с отдельным пользователем.

Резервное копирование: ежедневное, с выгрузкой на отдельный сервер (не на том же, где работает CRM).

База данных
Сложный пароль: для пользователя БД (не root).

Ограничение доступа: доступ к БД только с localhost (или через сокет).

Регулярные дампы: с шифрованием.

Хранение файлов (фото заказов)
Загрузка файлов: проверка расширения (jpg, png), MIME-типа, ограничение размера (5–10 МБ).

Хранение: за пределами веб-корня, доступ через защищённый эндпоинт.

Имена файлов: генерация случайного имени, избегать путей, содержащих ../.

Переменные окружения
Секреты в .env: никогда не хранить в коде. Файл .env не добавлять в репозиторий.

4.3 На этапе тестирования
Статический анализ кода: инструменты (SonarQube, bandit для Python, ESLint security плагины).

Тестирование безопасности: сканеры уязвимостей (OWASP ZAP, Nikto) для веб-приложения.

Пентест: ручная проверка на наличие XSS, SQLi, CSRF, уязвимостей авторизации.

Тестирование нагрузкой: проверка работы rate limiting, устойчивости к DoS.

Проверка прав доступа: попытка доступа к эндпоинтам с другой ролью.

4.4 На этапе эксплуатации и мониторинга
Мониторинг
Логи ошибок: централизованный сбор (например, Sentry).

Логи доступа: анализ подозрительных IP, частых неудачных логинов.

Оповещения: при падении сервиса, при превышении порога ошибок.

Управление уязвимостями
Регулярные обновления зависимостей: npm audit / pip-audit.

План реагирования на инциденты: что делать при утечке данных (уведомление клиентов, смена паролей).

Обучение сотрудников
Базовые правила безопасности: не передавать пароли, не открывать подозрительные ссылки, использовать сложные пароли.

5. Дополнительные рекомендации
5.1 Для локальной версии
Физическая безопасность компьютера: ограничение доступа к устройству.

Шифрование диска (BitLocker) на случай кражи.

Регулярное резервное копирование на внешний носитель.

5.2 Для облачной версии
Использовать VPS с минимальным набором служб.

Настроить автоматическое обновление ОС и ПО.

Включить логирование действий администратора сервера (SSH, sudo).

5.3 Соответствие 152-ФЗ
Если клиент ИП/ООО, он обязан соблюдать требования к обработке персональных данных.

Нужно:

Получать согласие на обработку ПД (можно в устной форме при приёме заказа, но лучше письменное или через клиентское приложение).

Обеспечить возможность удаления данных по запросу клиента.

Уведомлять Роскомнадзор о начале обработки (если это требуется).

6. Итоговый список проверок (краткий чек-лист)
Пароли хешированы bcrypt, сложные.

Ограничение попыток входа (5/15 минут).

Использование параметризованных запросов (ORM).

Валидация всех входных данных (типы, формат, длина).

Экранирование вывода (шаблоны).

CSP-заголовки.

CSRF-токены для форм (или JWT для API).

Rate limiting для API.

HTTPS включён, сертификат валидный.

Файрвол, закрыты лишние порты.

Переменные окружения, ключи не в коде.

Шифрование чувствительных полей в БД.

Регулярные обновления ОС, ПО, зависимостей.

Резервное копирование (ежедневное, на отдельный носитель).

Логирование действий пользователей и ошибок.

Тестирование на XSS, SQLi, CSRF.

Разделение прав (роли) на уровне API.

JWT с коротким сроком жизни.

Выполнение этих мер минимизирует риски и обеспечит надёжную защиту CRM от большинства типовых атак. Для особо критичных систем рекомендуется проводить регулярный аудит безопасности у сторонних специалистов.
