# Экран «Администрирование» — подробное описание

Экран «Администрирование» предназначен исключительно для пользователей с ролью **Администратор**. Здесь настраиваются параметры системы, управляются пользователи, определяются бизнес-правила (статусы заказов, пороги остатков, шаблоны уведомлений) и выполняется резервное копирование. Правильная конфигурация этого раздела обеспечивает гибкость системы и её адаптацию к реальным процессам сервисного центра.

---

## 1. Общая структура экрана

Экран «Администрирование» организован в виде набора вкладок (табов), каждая из которых отвечает за отдельную группу настроек. Все изменения сохраняются в базе данных и применяются без необходимости перезапуска системы.

### 1.1 Доступ
- Только администратор (роль `admin`).
- Если пользователь с другой ролью попытается перейти по URL – перенаправление на главную страницу с сообщением об отсутствии прав.

### 1.2 Элементы управления
- Кнопка «Сохранить» (внизу каждой вкладки) – применяет изменения.
- Кнопка «Отмена» – сбрасывает несохранённые изменения.
- Всплывающие уведомления об успешном сохранении или ошибках.

---

## 2. Вкладка «Пользователи»

Управление учётными записями сотрудников.

### 2.1 Список пользователей
- Таблица с колонками: **Имя**, **Email**, **Роль**, **Телефон**, **Дата регистрации**, **Последний вход**, **Статус** (активен / заблокирован).
- Фильтр по роли, поиск по имени/email.
- Кнопка **«+ Добавить пользователя»**.

### 2.2 Форма добавления / редактирования пользователя
- **Имя** (обязательно)
- **Email** (уникальный, формат email)
- **Телефон** (опционально)
- **Роль** (выбор из списка: `admin`, `manager`, `master`)
- **Пароль** (при создании – поле ввода; при редактировании – кнопка «Сбросить пароль» для отправки нового временного пароля на email)
- **Активен** (чекбокс; если снять – пользователь не сможет войти)

### 2.3 Действия
- **Редактировать** – открывает форму.
- **Сбросить пароль** – генерирует временный пароль, отправляет на email пользователя, просит сменить при первом входе.
- **Блокировать / Разблокировать** – меняет статус активности.

---

## 3. Вкладка «Настройки системы»

Общие параметры работы CRM.

### 3.1 Основные настройки
- **Название компании** (отображается в заголовке страницы и в уведомлениях).
- **Контактный телефон** (для подстановки в SMS-шаблоны).
- **Адрес сервисного центра** (для подстановки в SMS-шаблоны).
- **Рабочее время** (строка, например, «пн‑пт 9:00–19:00, сб 10:00–16:00»).
- **Формат номера заказа** (по умолчанию `MMDD-XXXX`; можно изменить префикс, например `MP-{дата}-{номер}`).
- **Количество дней для напоминания о профилактике** (по умолчанию 180 дней).

### 3.2 Настройки уведомлений
- **Включить SMS-уведомления** (глобальный переключатель).
- **Настройки SMS-провайдера**:
  - **API-ключ** (поле для ввода).
  - **Отправитель** (имя отправителя, которое увидит клиент).
  - **Тестовый режим** (не отправлять реальные SMS, только логировать).

### 3.3 Настройки интерфейса
- **Тема** (светлая / тёмная) – для пользователей (может быть переопределено в профиле).
- **Количество записей на странице** по умолчанию (20, 50, 100).

### 3.4 Настройки безопасности
- **Время жизни сессии** (в минутах).
- **Включить двухфакторную аутентификацию** (опционально, для администраторов).

---

## 4. Вкладка «Шаблоны уведомлений»

Настройка текстов SMS-сообщений для различных событий. Шаблоны поддерживают плейсхолдеры, которые заменяются реальными данными.

### 4.1 Список шаблонов
- **Заказ принят** – отправляется клиенту при создании заказа.
- **Статус изменён на «Готов»** – отправляется клиенту, когда заказ готов к выдаче.
- **Напоминание о профилактике** – отправляется клиенту через заданный интервал после завершения заказа.
- **Новый заказ (для сотрудников)** – оповещение менеджеров/мастеров (можно отдельно).
- **Товар ниже порога (администратору)**.

### 4.2 Форма редактирования шаблона
- **Название события** (только для справки).
- **Текст сообщения** (многострочное поле).
- **Плейсхолдеры** (справа – подсказка): `{client_name}`, `{order_number}`, `{device}`, `{company_name}`, `{address}`, `{work_hours}` и т.д.
- **Кнопка «Предпросмотр»** – показывает пример сообщения с подставленными данными.

---

## 5. Вкладка «Статусы заказов»

Управление статусами заказов и правилами переходов между ними.

### 5.1 Список статусов
- Таблица: **Название** (на русском), **Код** (для системы), **Цвет** (индикатор), **Порядок сортировки**.
- Возможность **добавить новый статус** (например, «Ожидание курьера»).

### 5.2 Редактирование статуса
- **Название** (обязательно).
- **Код** (уникальный, только латиница).
- **Цвет** (выбор из палитры).
- **Порядок** (для отображения в фильтрах).
- **Видимость для клиента** (если клиентское приложение, показывать ли этот статус).

### 5.3 Правила переходов
- Для каждого статуса можно задать, в какие статусы разрешён переход (и какой роли разрешено).
- Графическое отображение (диаграмма) – как опция.
- Пример: из «Принят» можно перейти в «Диагностика» (менеджер, мастер) или «Отказ» (администратор).
- По умолчанию должны быть настроены базовые переходы, описанные в ТЗ.

---

## 6. Вкладка «Пороги и склад»

Настройка параметров учёта товаров.

### 6.1 Пороги минимальных остатков
- **Глобальный порог** (если не задан для конкретного товара) – число.
- **Категорийные пороги** (например, для всех картриджей порог 2).

### 6.2 Настройка единиц измерения
- Возможность добавлять/редактировать единицы измерения (шт., кг, уп., комплект).

### 6.3 Прочие настройки склада
- **Включить учёт закупочных цен** (если нет – поля цен не показываются).
- **Включить автоматическое списание при добавлении материалов в заказ** (всегда должно быть включено).
- **Создавать уведомление при остатке ниже порога** (ежедневно).

---

## 7. Вкладка «Резервное копирование»

Инструменты для сохранения и восстановления данных.

### 7.1 Ручное резервное копирование
- Кнопка **«Создать бэкап»** – система создаёт дамп базы данных (SQL или архивирует все данные в JSON) и предлагает скачать файл.
- Указывается дата и время создания.

### 7.2 Восстановление из бэкапа
- Поле загрузки файла (только файлы с расширением `.sql` или `.zip`).
- **Важно:** перед восстановлением система предлагает подтверждение и предупреждает, что все текущие данные будут заменены. Рекомендуется предварительно создать бэкап.

### 7.3 Автоматическое резервное копирование
- **Включить автоматическое** (чекбокс).
- **Периодичность**: ежедневно / еженедельно.
- **Время** (по умолчанию 03:00).
- **Хранить копии** (количество последних копий, например 7).
- **Место хранения** (локальная папка на сервере или облачное хранилище – настраивается).

---

## 8. Вкладка «Логирование»

Просмотр и фильтрация системных логов действий пользователей.

### 8.1 Список логов
- Таблица с колонками: **Дата и время**, **Пользователь**, **Действие** (текстовое описание), **Сущность** (например, «Заказ 2403-0012»), **Изменения** (что было изменено).
- Фильтры: по дате (диапазон), по пользователю, по типу действия (создание, изменение, удаление, смена статуса и т.д.).
- Поиск по тексту действия или номеру заказа.

### 8.2 Детали
- При клике на запись – подробная информация (JSON-представление изменений, если доступно).
- Возможность экспорта логов в XLSX (за выбранный период).

---

## 9. Вкладка «Настройки интерфейса»

Кастомизация внешнего вида системы для разных ролей (опционально).

### 9.1 Настройки для ролей
- Для каждой роли можно задать:
  - **Какие вкладки видны** (например, менеджеру – не показывать склад и отчёты по финансам).
  - **Какие поля отображать в списках** (например, для мастера – не показывать колонку стоимости).
- Это позволяет упростить интерфейс для разных категорий сотрудников.

## 10. Технические детали

### 10.1 API-эндпоинты
- `GET /api/admin/users` – список пользователей.
- `POST /api/admin/users` – создание.
- `PUT /api/admin/users/:id` – обновление.
- `DELETE /api/admin/users/:id` – удаление.
- `POST /api/admin/users/:id/reset-password` – сброс пароля.
- `GET /api/admin/settings` – получение всех настроек.
- `PUT /api/admin/settings` – обновление.
- `GET /api/admin/sms-templates` – шаблоны.
- `PUT /api/admin/sms-templates` – обновление.
- `GET /api/admin/statuses` – список статусов.
- `POST /api/admin/statuses` – создание.
- `PUT /api/admin/statuses/:id` – обновление.
- `DELETE /api/admin/statuses/:id` – удаление (если не используется).
- `GET /api/admin/statuses/transitions` – правила переходов.
- `PUT /api/admin/statuses/transitions` – обновление правил.
- `POST /api/admin/backup` – создание бэкапа.
- `POST /api/admin/restore` – восстановление.
- `GET /api/admin/logs` – логи (с фильтрацией).

### 10.2 База данных – дополнения
- Таблица `settings` (ключ-значение) – для хранения всех настроек.
- Таблица `notification_templates` (id, type, template).
- Таблица `statuses` (id, code, name, color, order, visible_to_client).
- Таблица `status_transitions` (from_status, to_status, allowed_roles).
- Таблица `user_activity_logs` (id, user_id, action, entity_type, entity_id, changes, created_at).
