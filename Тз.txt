# Техническое задание на разработку CRM-системы для сервисного центра «Мастер Принт»

## 1. Общие сведения

### 1.1 Наименование проекта
CRM-система для учёта заказов, клиентов, складских запасов и автоматизации уведомлений (далее – Система).

### 1.2 Цели создания Системы
- Автоматизация учёта заказов на ремонт и обслуживание оргтехники.
- Централизованное хранение информации о клиентах и истории обращений.
- Контроль статусов выполнения заказов в реальном времени.
- Учёт расходных материалов и запчастей на складе.
- Автоматическое информирование клиентов и сотрудников о событиях через SMS-уведомления.
- Повышение прозрачности бизнеса и снижение операционных издержек.

### 1.3 Заказчик
ИП [Название], сервисный центр «Мастер Принт».

### 1.4 Разработчик
[Данные разработчика].

## 2. Термины и сокращения

- **Заказ** – единица работы по ремонту/обслуживанию устройства.
- **Клиент** – физическое или юридическое лицо, обратившееся в сервис.
- **Статус заказа** – этап выполнения заказа (Принят, Диагностика, Согласование, Ожидание запчастей, Ремонт, Готов, Выдан).
- **Расходный материал** – товары, используемые при ремонте (картриджи, тонер, бумага, запчасти).
- **SMS-уведомление** – сообщение, доставляемое на телефон клиента или сотрудника через SMS-шлюз.
- **Роль** – набор прав пользователя в Системе (администратор, мастер, менеджер).

---

## 3. Роли пользователей

| Роль | Описание |
|------|----------|
| **Администратор** | Полный доступ ко всем функциям: управление заказами, клиентами, складом, пользователями, настройками, просмотр аналитики. |
| **Мастер** | Работает с заказами: просмотр назначенных заказов, изменение статусов, добавление комментариев и использованных материалов. Не имеет доступа к финансовой отчётности и управлению пользователями. |
| **Менеджер** | Приём заказов, работа с клиентами, изменение статусов до «Согласование» и «Готов», просмотр базы клиентов, создание напоминаний. Может видеть отчёты по заказам. |

## 4. Функциональные требования

### 4.1 Модуль «Электронный журнал заказов»

#### 4.1.1 Создание нового заказа
Форма создания заказа содержит следующие поля:
- **Клиент** (выбор существующего или создание нового)
- **Устройство** (текстовое поле, обязательное)
- **Модель** (текстовое поле)
- **Серийный номер** (текстовое поле, опционально)
- **Неисправность** (описание, многострочное поле, обязательное)
- **Комплектация** (чекбоксы: блок питания, кабель USB, кабель питания, мышь, клавиатура, картридж; возможность добавить свой текст)
- **Дата приёма** (по умолчанию текущая, можно изменить)
- **Желаемая дата готовности** (опционально)
- **Предварительная стоимость** (числовое поле)
- **Принявший сотрудник** (выбор из списка мастеров/менеджеров, автоматически текущий пользователь)

После сохранения заказа:
- Присваивается уникальный номер заказа (формат: `MMDD-XXXX`, где MMDD – дата, XXXX – порядковый номер за день).
- Статус устанавливается «Принят».
- В карточке клиента появляется ссылка на новый заказ.

#### 4.1.2 Просмотр и редактирование заказа
- Отображение всех полей заказа.
- Возможность редактирования в зависимости от роли (администратор может редактировать всё, мастер – только статус, использованные материалы и комментарии).
- Журнал изменений: фиксация всех изменений статуса и комментариев с указанием автора и времени.

#### 4.1.3 Список заказов
- Таблица с фильтрами:
  - по статусу
  - по дате приёма (диапазон)
  - по клиенту
  - по мастеру
- Возможность сортировки по любому столбцу.
- Поиск по номеру заказа, телефону клиента, устройству.

#### 4.1.4 Статусная модель
Переходы между статусами должны быть логически ограничены (например, из «Готов» нельзя вернуться в «Ремонт» без специального разрешения администратора).  
Допустимые переходы:
- Принят → Диагностика
- Диагностика → Согласование, Принят (если ошибка)
- Согласование → Ожидание запчастей, Ремонт, Принят (при отказе клиента – особая отметка)
- Ожидание запчастей → Согласование (запчасти пришли), Принят (отказ)
- Ремонт → Готов
- Готов → Выдан
- Выдан (финальный, изменения запрещены)

При переходе в статус «Готов» автоматически инициируется отправка SMS-уведомления клиенту.

### 4.2 Модуль «База клиентов»

#### 4.2.1 Карточка клиента
- **Тип клиента** (частное лицо / организация)
- **ФИО / Название организации**
- **Телефон** (обязательное, уникальное)
- **Email** (опционально)
- **Адрес** (опционально)
- **Комментарий** (многострочное поле)
- **Дата первого обращения**
- **История заказов** (список всех заказов клиента со статусами и ссылками)
- **Теги** (например, «VIP», «Постоянный», «Оптовик»)

#### 4.2.2 Поиск и фильтрация
- Быстрый поиск по телефону, имени, email.
- Расширенный поиск по тегам, дате последнего заказа.

#### 4.2.3 Действия с клиентом
- Создать новый заказ (переход к форме заказа с предзаполненными данными клиента).
- Добавить заметку.
- Отправить SMS-уведомление (массовая рассылка для сегмента клиентов – только администратор).

### 4.3 Модуль «Складской учёт»

#### 4.3.1 Сущности
- **Категория** (например, «Картриджи», «Тонер», «Бумага», «Запчасти»)
- **Товар**:
  - Наименование
  - Категория
  - Артикул (уникальный)
  - Единица измерения (шт., кг, уп.)
  - Цена закупочная (опционально)
  - Цена продажи (для розницы, опционально)
  - Минимальный остаток (порог для уведомления)

#### 4.3.2 Поступление товара
- Выбор товара
- Количество
- Дата поступления
- Склад (если несколько)

#### 4.3.3 Списание товара
- Автоматическое списание при использовании в заказе (мастер отмечает, какие товары и в каком количестве израсходованы).
- Ручное списание (при порче, утере, инвентаризации) с комментарием.

#### 4.3.4 Отчёт по остаткам
- Текущие остатки с подсветкой товаров, достигших минимального порога.
- История движения по каждому товару.

### 4.4 Модуль «Уведомления (SMS)»

#### 4.4.1 Типы уведомлений
- **Для клиента**:
  - Заказ принят (информационное SMS).
  - Статус изменён на «Готов» (SMS с текстом о готовности и адресом).
  - Напоминание о профилактике через заданный интервал (SMS, настраивается администратором).
- **Для сотрудников**:
  - Новый заказ (SMS для мастеров/менеджеров или выбранной группы).
  - Заказ требует согласования с клиентом (SMS менеджеру).
  - Товар на складе ниже минимума (SMS администратору).

#### 4.4.2 Техническая реализация
- Использование SMS-шлюза для отправки SMS-уведомлений клиентам и сотрудникам.

#### 4.4.3 Настройка уведомлений
- В профиле пользователя (клиента или сотрудника) возможность включить/отключить определённые типы SMS-уведомлений.

### 4.5 Модуль «Аналитика и отчёты»

#### 4.5.1 Основные метрики на дашборде (для администратора)
- Количество заказов сегодня / неделя / месяц
- Среднее время выполнения заказа
- Распределение заказов по статусам (круговая диаграмма)
- Топ-5 популярных услуг (по категориям неисправностей)
- Топ-5 клиентов по количеству заказов
- Выручка за период (по завершённым заказам, если введена стоимость)
- Остатки ниже порога (список товаров)

#### 4.5.2 Отчёты (с возможностью экспорта в XLSX)
- Отчёт по заказам за период (с фильтрами по статусам, мастерам).
- Отчёт по движению товаров.
- Финансовый отчёт (доходы/расходы по заказам; обязателен к реализации).

### 4.6 Модуль «Администрирование»

#### 4.6.1 Управление пользователями
- Добавление/редактирование/удаление сотрудников.
- Назначение роли (администратор, мастер, менеджер).
- Сброс пароля.

#### 4.6.2 Настройки системы
- Настройка текстов уведомлений (шаблоны push-сообщений).
- Настройка порогов минимальных остатков.
- Управление статусами и их последовательностью (возможность добавления кастомных статусов).
- Резервное копирование данных.

#### 4.6.3 Логирование действий
- Фиксация всех значимых действий (создание/изменение заказа, изменение статуса, списание товара) с указанием пользователя, времени, сути изменения.

## 5. Требования к интерфейсу (UI/UX)

- **Платформа**: кроссплатформенное веб-приложение (адаптивное под мобильные устройства) для сотрудников.
- **Основные принципы**:
  - Минималистичный дизайн, ориентированный на быстрый ввод данных.
  - Все основные действия доступны в 2-3 клика.
  - Цветовое кодирование статусов (например, красный – просрочено, жёлтый – ожидание, зелёный – готов).
  - Поддержка тёмной темы (опционально).

- **Экраны веб-версии (для сотрудников):**
  - Дашборд (сводка ключевых показателей).
  - Список заказов с фильтрами.
  - Карточка заказа.
  - Список клиентов.
  - Карточка клиента.
  - Склад (список товаров, остатки, форма поступления/списания).
  - Отчёты.
  - Настройки.

## 7. Требования к безопасности и производительности

- **Аутентификация**:
  - Для веб-приложения: вход по email/паролю с использованием JWT (или сессий).
  - Для мобильного приложения клиента: вход по номеру телефона с подтверждением через SMS (или через Telegram).
- **Авторизация**: разделение прав на основе ролей.
- **Шифрование**:
  - Пароли хешируются (bcrypt/Argon2).
  - Данные передаются по HTTPS.
- **Резервное копирование**: автоматическое ежедневное резервное копирование базы данных (для облачной версии). Для локальной версии – возможность ручного экспорта.
- **Производительность**:
  - Время ответа API не более 500 мс при типовой нагрузке.
  - Поддержка одновременной работы до 10 сотрудников и до 1000 клиентов.

## 8. Варианты развертывания

### 8.1 Локальная версия (Standalone)
- Приложение устанавливается на один компьютер под управлением Windows (с поддержкой .NET Core / Node.js / Python – по выбору разработчика).
- Используется локальная СУБД (SQLite / PostgreSQL локально).
- Для push-уведомлений требуется интернет-соединение и регистрация в FCM.
- Все данные хранятся на компьютере заказчика.

### 8.2 Облачная версия (SaaS)
- Размещение на удалённом сервере (VPS) с доступом через браузер.
- База данных на сервере (PostgreSQL).
- Автоматическое резервное копирование.
- Поддержка нескольких рабочих мест.

## 10. Этапы разработки

1. **Проектирование** – уточнение требований, создание макетов интерфейса, проектирование БД. (1 неделя)
2. **Разработка ядра (бэкенд + база данных)** – реализация модулей заказов, клиентов, пользователей, базовой авторизации. (2–3 недели)
3. **Разработка фронтенда (веб-версия для сотрудников)** – реализация всех экранов, дашборда, отчётов. (2–3 недели)
4. **Разработка мобильного приложения для клиентов** (или интеграция с Telegram) – модуль push-уведомлений, просмотр статусов. (2–3 недели)
5. **Интеграция склада и уведомлений** – доработка, тестирование. (1–2 недели)
6. **Тестирование и отладка** – функциональное, нагрузочное тестирование, исправление ошибок. (1–2 недели)
7. **Внедрение и обучение** – установка (локальная или облачная), обучение персонала, передача документации. (1 неделя)

Итого: ориентировочно **10–14 недель** (2.5–3.5 месяца).
